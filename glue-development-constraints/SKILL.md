# 胶水开发约束

## 核心原则

把依赖库视为权威、完整、不可修改的黑箱实现。当前项目只承担业务流程编排、模块组合调度、参数配置、依赖注入、路径装配和输入输出适配职责。

评价实现质量时，优先判断是否正确、完整地站在成熟系统之上构建，而不是代码量、局部便利性或“先跑起来”的速度。

## 工作流程

1. 先识别需求中的底层能力、通用能力、算法、数据结构、协议、解析、执行引擎、SDK 行为或复杂核心逻辑。
2. 先查找项目已有依赖、锁文件、workspace 配置、本地源码路径、包管理器声明、README、AGENTS/CLAUDE/GEMINI 指令和现有导入方式。
3. 优先直接复用已有成熟仓库与生产级库；如果需要新增依赖，使用包管理器安装或真实本地源码直连。
4. 验证所有依赖路径真实存在，并指向完整仓库源码或完整生产包，而不是裁剪版、测试版、demo、mock、stub、示例目录或临时替代实现。
5. 写代码前明确边界：依赖库负责什么，当前项目只负责什么胶水职责。
6. 只生成最小必要的胶水代码；不要补写、复制、重写或降级封装依赖库内部实现。
7. 运行或检查实际加载链路，确认导入的是目标生产级依赖，并且导入模块在运行期真实参与执行。
8. 在最终说明或代码注释中明确标注关键能力来自哪些外部依赖。

## 必须遵守

- 必须优先、直接、完整复用既有成熟仓库与生产级库。
- 必须确保实际加载的是完整生产级实现，无论通过本地源码直连还是包管理器安装。
- 必须让所有依赖路径真实存在，并指向目标完整仓库源码或完整生产包。
- 必须直接导入完整依赖模块，除非依赖库官方 API 本身要求从子模块导入。
- 必须确保 `sys.path`、workspace resolution、包管理器别名、依赖注入链路和运行时入口加载的是目标生产级本地库或生产包。
- 必须确认所有导入模块在运行期真实参与执行，避免只导入不用的伪集成。
- 必须把当前项目限制在业务流程编排、模块组合调度、参数配置与输入输出适配。
- 必须在生成代码、实现说明或审查结论中标明哪些功能来自外部依赖。

## 严禁行为

- 不得自行实现底层或通用逻辑。
- 不得为了方便复制依赖库代码到当前项目后修改使用。
- 不得对依赖库进行功能裁剪、逻辑重写、降级封装或替代实现。
- 不得使用简化版、替代版、重写版依赖冒充真实库实现。
- 不得通过路径遮蔽、重名模块、隐式 fallback 或环境变量误配加载非目标实现。
- 不得在当前项目中实现依赖库已提供的同类功能。
- 不得使用 mock、stub、demo、placeholder、空逻辑或“先写接口后补实现”的代码作为真实能力来源。
- 不得重复实现算法、数据结构或复杂核心逻辑。
- 不得将依赖库中的复杂逻辑拆出后自行实现。
- 不得在任何情况下生成或补写依赖库内部实现代码。
- 不得把外部依赖包一层层包成只暴露子集能力的二次抽象；只允许做必要的参数适配、错误传播、生命周期接线和 I/O 转换。

## 允许的胶水代码

- 配置依赖库初始化参数、凭据来源、路径、环境变量和运行模式。
- 组合多个成熟库的调用顺序，处理业务流程状态和输入输出映射。
- 把项目数据结构转换为依赖库公开 API 需要的参数格式，再把依赖库返回值转换为业务层输出。
- 做薄层错误处理、日志、重试策略和资源释放，但不要吞掉依赖库语义或重写依赖库策略。
- 写测试时可以 mock 项目边界 I/O，但不能用 mock/stub 冒充依赖库核心能力的生产实现。

## 集成验证清单

完成前逐项检查：

- 目标依赖是否来自完整生产级库或完整本地仓库源码。
- 锁文件、包管理器配置、workspace 配置、`PYTHONPATH`/`sys.path`、Node resolution、构建别名或运行入口是否会加载正确目标。
- 当前项目是否没有同类算法、数据结构、协议解析、执行引擎或 SDK 逻辑的重复实现。
- 是否不存在本地同名模块、fallback 分支、demo 目录、test fixture、mock/stub 或简化实现被误加载。
- 所有导入是否都真实参与运行期执行。
- 新增代码是否仅为业务编排、组合调度、参数配置或输入输出适配。
- 最终说明是否标明外部依赖承担的真实能力。

## 遇到缺失依赖时

如果找不到真实生产级依赖，不要写替代实现。先报告缺口，说明需要安装的包、需要直连的仓库路径或需要用户提供的完整依赖源码；只有在依赖可用并且加载链路可验证后再继续实现。
